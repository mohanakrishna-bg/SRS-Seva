# Phase 2.1: Accounting Module Enhancements - Technical Research

This document outlines the technical findings, database structures, integration routes, and codebase analysis required to implement the features specified in `02.1-CONTEXT.md` for **Phase 2.1**.

---

## 1. Codebase Analysis & Data Models

Our double-entry accounting schema is located in `backend/app/models/accounting.py` and uses four primary models:

### A. Existing Database Models (`models/accounting.py`)
1. **`AccountHead`**: Chart of accounts featuring hierarchical child-parent keys (`ParentId`). Types supported: `'Income'`, `'Expense'`, `'Asset'`, `'Liability'`.
2. **`JournalEntry`**: High-level double-entry voucher record (`EntryDate`, `Narration`, `SourceModule`, `SourceRefId`).
3. **`JournalLine`**: Transaction debit/credit rows (`JournalEntryId`, `AccountId`, `Debit`, `Credit`).
4. **`BankTransaction`**: Cash/Bank records mapped to bank accounts. Currently has a boolean `IsReconciled` flag and optional `JournalEntryId`.

### B. Findings on Balance Reconciliation (`FIN-02`)
In the current implementation of `create_bank_transaction` (`backend/app/api/accounting.py`), logging a new transaction instantly alters `bank_acc.CurrentBalance` regardless of the reconciliation state.
*   **The Bug**: Uncleared or bounced cheque entries directly mutate the main operating balance.
*   **Proposed Fix**:
    1. Update `create_bank_transaction` to only alter `bank_acc.CurrentBalance` if `IsReconciled` is explicitly passed as `True`.
    2. Revamp the `/bank-transactions/{id}/reconcile` POST router:
        - When transitioning `IsReconciled` to `True` (Cleared): add/subtract the `Amount` from `bank_acc.CurrentBalance`.
        - When transitioning `IsReconciled` to `False` (Uncleared): reverse the calculation.
    3. Update schemas in `backend/app/schemas/accounting.py` to allow returning reconciliation balances.

## 2. Granular Expense Capture (`FIN-03`)

To enable granular expense vouchers, we will map expenditures directly to balanced double-entry journal items:

### A. Category Selection
*   We will leverage the hierarchical `AccountHead` schema. We will bootstrap standard operational expense heads in the seed database or master table (e.g. `EX01` (Alankara/Flowers), `EX02` (Prasada Ingredients), `EX03` (Electricity/Admin), `EX04` (Salary)).
*   A new API endpoint `@router.get("/accounts/expenses")` will filter and return all expense-type account heads.

### B. Expense Voucher Post Logic
When saving a Payment Voucher, the backend will validate the double entry and automatically post a balanced `JournalEntry` under `SourceModule="Inventory"` or `"Manual"`:
*   **Debit**: Selected Expense Head Account (e.g., `EX02` - Prasada Ingredients).
*   **Credit**: Cash Account (`A001`) or Selected Bank Asset Account (`A002`).

## 3. Daily Day-Closing & Audit Locks (`FIN-03`)

To prevent historic data manipulation after books have been audited and signed off by the administrator, we will implement date-level locks.

### A. Database Migration
Add a new model to track closed dates in `backend/app/models/accounting.py`:
```python
class ClosedDay(Base):
    __tablename__ = "ClosedDay"
    Date = Column(String, primary_key=True) # YYYY-MM-DD
    ClosedAt = Column(DateTime, default=datetime.datetime.utcnow)
```

### B. Enforcing the Lock in API Routes
Add a dependency or helper function `check_date_locked(entry_date: str, db: Session)` in backend routers. If `entry_date` exists in `ClosedDay`:
*   Block all `POST`, `PUT`, and `DELETE` requests on transaction, receipt, and ledger routes for that date with `403 Forbidden` ("This date's ledger has been locked and signed off").
*   Enforce a **2-hour rolling grace period** for standard staff roles (checked by evaluating `CreatedAt` on the target record). Admin roles bypass the grace-period lock but respect the explicit daily `ClosedDay` lock.

## 4. UI Layout & Print Architecture (`REP-02`)

### A. Print Optimization
We will inject a CSS print stylesheet standard in `index.css`:
```css
@media print {
  body {
    background: white !important;
    color: black !important;
  }
  .sidebar, .nav-bar, .toolbar-actions, button {
    display: none !important;
  }
  .printable-area {
    width: 100% !important;
    margin: 0 !important;
    padding: 0 !important;
    border: none !important;
  }
}
```
This enables clean print outputs when clicking "Print" natively in the browser on the report statements page.

### B. Drill-Down Navigation
*   **Sub-ledger statement**: A new overlay/modal will render in `AccountingPage.tsx`. When an Account Head is clicked, it calls `accountingApi.listJournal({ account_id: id })` to load only lines matching that account in a chronological ledger card.
*   **Devotee details**: Connecting row clicks in `CollectionDashboard.tsx` to launch devotee detail popovers.
